# PR

`origin` (em3s/actionbase)에만 한국어 PR을 생성/업데이트한다.

**절대 upstream (kakao/actionbase)에 PR을 만들지 않는다.** upstream 기여는 `patch-upstream`을 사용한다.

**모든 단계에서 사용자 승인을 받은 후 진행한다.**

## Workflow

### Step 1: 기존 PR 확인

```bash
git branch --show-current
gh pr view --json number,title,body,url 2>/dev/null
```

사용자에게 보고:
- 현재 브랜치명
- **PR 있음:** `#번호`, 제목, URL 표시
- **PR 없음:** "이 브랜치에 기존 PR 없음"

**사용자 확인 후 진행.**

### Step 2: Diff 분석

```bash
git diff main...HEAD --stat
git diff main...HEAD
git log main...HEAD --oneline
```

### Step 3: 초안 작성

diff를 기반으로 아래 스타일 가이드에 따라 PR 제목과 본문을 **한국어로** 작성한다.

**기존 PR이 있는 경우:** Step 1의 제목/본문을 출발점으로 사용. 여전히 정확한 내용은 유지하고, 새 변경사항을 반영하며, 오래된 내용은 제거한다.

**아직 생성/업데이트하지 않는다.** 먼저 사용자에게 한국어 초안을 보여준다.

**사용자 승인 또는 수정 요청을 기다린다.**

### Step 4: 실행

사용자가 명시적으로 승인한 후에만 실행:

**새로 생성하는 경우:**
```bash
git push -u origin HEAD
gh pr create --title "..." --body "..."
```

**업데이트하는 경우:**
```bash
gh pr edit <number> --title "..." --body "..."
```

**PR 본문에 "Generated with Claude Code" 등 AI 귀속 푸터를 절대 넣지 않는다.**

완료 후 PR URL을 반환한다.

## PR Style Guide

[criccomini (Chris Riccomini)](https://github.com/slatedb/slatedb/pulls?q=is%3Apr+author%3Acriccomini) — SlateDB 메인테이너 스타일 참고.

### Template

```markdown
## 요약
## 변경사항
## 리뷰어 참고
## 체크리스트
```

### 요약 — 이 PR이 존재하는 이유

- **첫 문장**: 이 PR이 무엇을 하는지 (한 줄)
- **배경**: 동기, 관련 이슈, RFC, 이전 PR
- 버그 수정: 장애 시나리오를 구체적으로 기술 (타임라인, 레이스 컨디션, 에러 로그)
- 성능: 전후 수치 포함 (`10-20ms → 1.5ms`)
- 이슈 링크: `Fixes #N` / `Closes #N`

### 변경사항 — 무엇을 바꿨는가

- 불릿 리스트, 3-6개 항목
- 동사로 시작: 추가, 수정, 교체, 제거, 개선
- 파일명, 함수명, 타입명은 백틱 사용

### 리뷰어 참고 — 리뷰어를 돕는 섹션

가장 가치 있는 섹션:

- **검토한 대안**: 거절한 접근법과 이유
- **리뷰 순서**: 커밋별 리뷰가 도움될 때 제안
- **주의사항**: 느린 테스트, 알려진 제한, 향후 작업
- **솔직한 톤**: 어색한 부분, 완벽하지 않은 부분, 향후 PR로 미룬 부분
- **사소한 PR**: `없음.`으로 충분

### 체크리스트

```markdown
- [ ] 작고 집중된 PR (테스트 제외 500줄 이하); 또는 분할 계획이 있는 Draft
- [ ] 관련 이슈 링크 또는 설명에 맥락 추가
- [ ] diff 셀프 리뷰 완료; 까다로운 부분에 코멘트 추가
- [ ] 테스트 추가/수정 및 로컬에서 통과 확인
- [ ] 빌드 및 린트 체크 실행
- [ ] 브레이킹 체인지 명시 및 마이그레이션 노트
- [ ] 성능 영향 검토; 필요시 벤치마크 추가
```

체크리스트는 솔직하게 — 해당 안 되면 체크하지 않는다.

### PR 크기별 가이드

| PR 크기 | 요약 | 변경사항 | 리뷰어 참고 | 체크리스트 |
|---------|------|---------|------------|-----------|
| Large (500+ lines) | 전체 맥락, 대안, 향후 목표 | 4-6개 | 상세히 | 전체 |
| Medium | 설계 트레이드오프, 근거 | 3-4개 | 핵심만 | 전체 |
| Small (< 50 lines) | 2-3문장 | 1-2개 | 생략 또는 `없음.` | 생략 가능 |

### 원칙

1. **"왜 이 방법"과 "왜 다른 방법은 안 되는지"** 모두 설명
2. **근거 첨부** — 에러 로그, 벤치마크 수치, GitHub Actions URL
3. **리뷰어 시간 절약** — 리뷰 순서 제안, 까다로운 부분 표시, 예상 질문에 미리 답변
4. **솔직하게** — 한계, 미완성 부분, vibe-coded 구간 인정
5. **감사 인사** — `리뷰 감사합니다!`
