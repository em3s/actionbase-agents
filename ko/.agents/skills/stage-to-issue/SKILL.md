# Stage-to-Issue 명령어

구현 중 새로운 작업이 발견되면 GitHub 이슈로 기록하고 현재 작업으로 돌아갑니다.

## 사용법

- `stage-to-issue Optimize CodeQL to run only on changed paths`
- `stage-to-issue Query engine needs cursor reset on schema change`

## 에이전트 서명

이슈 본문에 CLI와 모델을 포함하세요.
형식: `<cli> (<model>)` -- 예: `claude code (opus 4.6)`, `codex cli (o3)`

## 단계

1. **컨텍스트 수집** -- 대화에서 문제, 동기, 관련 세부 사항 파악
2. **관련 이슈 확인** -- 현재 작업 중인 이슈/PR이 있으면 참조로 포함
3. **이슈 초안 작성** -- 제목과 본문을 한국어로 작성
4. **초안 제시** -- 사용자에게 이슈를 보여줌
5. **중단** -- 사용자 확인 대기
6. **생성** -- `gh issue create --title "<title>" --body "<body>"`
7. **확인** -- 생성된 이슈 번호와 URL 표시
8. **복귀** -- 현재 작업 재개

## 이슈 본문 템플릿

```
## 문제
<무엇을 해야 하는지, 왜 필요한지>

## 맥락
<현재 대화에서 파악한 관련 세부사항>

## 관련
- #<현재-작업-이슈-번호> 작업 중 발견 (해당 시)

## 접근 방향
<해결 방법이 있으면 제안, 없으면 이 섹션 생략>

---
Created by **<agent signature>**
```

## 규칙

- 사용자 확인 없이 이슈를 절대 생성하지 않음
- 간결하게 유지 -- 사이드 작업이지 사양서가 아님
- 다른 세션/에이전트가 독립적으로 작업할 수 있을 만큼의 컨텍스트 포함
- 현재 작업 중인 이슈가 있으면 Related 섹션에 참조
- **이슈에 서명** -- 하단에 에이전트 서명 포함
- 모든 출력 (제목, 본문)은 한국어로 작성
- 생성 후 즉시 현재 작업으로 복귀 -- 새 이슈 작업을 시작하지 않음
