# Plan 명령어

새로운 요구사항을 분석하고 구현 계획을 GitHub 이슈로 만듭니다.

## 사용법

- `/plan Add a new schema type for bookmarks`
- `/plan Refactor query engine to support cursor-based pagination`

## 에이전트 서명

이슈 본문에 CLI와 모델을 포함하세요.
형식: `<cli> (<model>)` -- 예: `claude code (opus 4.6)`, `opencode (codex 5.2)`

## 단계

1. **요구사항 분석** -- 요청을 명확한 용어로 재정리
2. **코드베이스 탐색** -- 관련 파일, 의존성, 영향 범위 파악
3. **구현 계획 수립** -- 단계별로 분해, 위험 식별
4. **이슈 초안 제시** -- 사용자에게 보여줌
5. **중단** -- 사용자 확인 대기
6. **이슈 생성** -- `gh issue create`로 GitHub 이슈 생성
7. **다음 단계 안내** -- `/implement <issue>` 또는 세션 종료

## 이슈 본문 템플릿

```
## 요약
<무엇을 해야 하는지, 왜 필요한지>

## 분석
<영향받는 파일, 의존성, 영향 범위>

## 계획
- [ ] 1단계: <설명>
- [ ] 2단계: <설명>
- [ ] N단계: <설명>

## 위험 요소
- <위험 수준>: <설명>

## 예상 복잡도
<상/중/하>

---
Created by **<agent signature>**
```

## 세션 후 흐름

이슈 생성 후 두 가지 경로:
- **계속 개발**: `/implement <issue>` 로 바로 구현 시작
- **나중에 개발**: 세션 종료, 다음에 `/implement <issue>`로 시작

## 규칙

- 사용자 확인 없이 이슈를 절대 생성하지 않음
- 이슈에 모든 컨텍스트를 담아야 함 -- 다른 세션/에이전트가 독립적으로 작업 가능해야 함
- 코드베이스 탐색 결과를 이슈에 포함 (관련 파일 경로, 기존 패턴)
- **이슈에 서명** -- 하단에 에이전트 서명 포함
- 모든 출력은 한국어로 작성 (코드와 코드 내 주석은 영어)
