# Implement 명령어

이슈를 받아 구현합니다. 브랜치 → PR → 구현 → 빌드 → 리뷰 → 반복.

## 사용법

- `/implement 185`
- `/implement https://github.com/kakao/actionbase/issues/185`

## 에이전트 서명

모든 작업에 수행한 에이전트와 모델을 포함해야 합니다.
형식: `<cli> (<model>)` -- 예: `claude code (opus 4.6)`, `opencode (codex 5.2)`

세션 시작 시 자신의 CLI와 모델을 식별하고 일관되게 사용해야 합니다.

## 단계

1. **조회** -- `gh issue view <arg>` (이슈 본문에서 Plan 확인)
2. **브랜치** -- 현재 브랜치에서 `feat/issue-<number>-<short-desc>` 생성
3. **빈 커밋 + PR** -- 아래 템플릿을 사용하여 PR 본문에 계획 작성
4. **중단** -- 사용자 확인 대기
5. **구현** -- 각 단계를 코딩하고, 커밋, 푸시하고 PR 본문에서 `[x]` 체크
6. **빌드 + 테스트** -- 빌드와 테스트를 실행하여 검증
7. **리뷰** -- `/code-review` 실행, 결과를 사용자에게 제시
8. **코멘트** -- `## Review Round N`으로 PR 코멘트에 기록
9. **수정** -- 합의된 변경 사항 적용, 커밋, 푸시
10. **반복** -- CRITICAL/HIGH 이슈가 없을 때까지 6-9단계 반복

## PR 본문 템플릿

```
## 요약
<설명>

Closes #<number>

## 계획
Created by **<agent signature>**

- [ ] 1단계
- [ ] 2단계
- [ ] N단계

## 진행 상황
<!-- 각 에이전트는 여기에 추가만 합니다. 이전 항목을 절대 삭제하지 마세요. -->
- [x] 1단계 — **<agent signature>** (commit abc1234)
```

## 리뷰 코멘트 템플릿

```
## 리뷰 라운드 N
by **<agent signature>**

### 발견 및 수정 사항

| 심각도 | 이슈 | 상태 |
|--------|------|------|
| CRITICAL | <설명> | 수정: <내용> |
| HIGH | <설명> | 수정 / 미수정: <사유> |
| WARNING | <설명> | 수정: <내용> |
| SUGGESTION | <설명> | 수정 / 건너뜀: <사유> |

### 결과
모든 테스트 통과. CRITICAL/HIGH 이슈 없음.
```

심각도 수준: CRITICAL > HIGH > WARNING > SUGGESTION.
건너뛴 항목 포함 모든 이슈를 테이블에 기록합니다 (`Skipped: <reason>`으로 표시).

## 규칙

- 사용자 확인 없이 절대 진행하지 않음
- 커밋은 작고 집중적으로 유지
- PR 본문을 항상 최신 진행 상황으로 업데이트
- **모든 작업에 서명** -- PR 생성, 계획 업데이트, 단계 완료, 리뷰 코멘트에 에이전트 서명 포함
- **이력 보존** -- PR 본문 업데이트 시 다른 에이전트의 항목을 절대 삭제하지 않음. 추가만 허용.
- 모든 리뷰 논의를 라운드 번호와 함께 PR 코멘트로 기록
- 모든 출력 (PR 제목, 본문, 커밋, 코멘트)은 한국어로 작성 (코드와 코드 내 주석은 영어)
